- **D-0011** (two-way ADR, 2026-06-18): **Dispose terminality.** `query.dispose` of an ACTIVE query (Running/CancelRequested, no complete yet) now yields exactly one `query.complete` with `status:"disposed"`, strengthening I2 to "every accepted query emits exactly one terminal" (the prior dispose-exemption is removed). Core adds a `disposing` phase: dispose replies `{}` and emits `driver.queryDispose` but holds `ActiveQueryId` and emits no terminal until the runner confirms the pump stopped; the `driver.queryDispose` result then emits the single `query.complete(disposed)` and releases the connection. The runner awaits the streaming pump task before acking dispose, so the connection is never freed while the old reader is still unwinding (R009) and a new query cannot race the old one on the same session. Driver events during `disposing` are suppressed (no double terminal). Chosen over keeping dispose silent so the viewer/client can rely on one pinned terminal per accepted query.
- **D-0012** (two-way ADR, 2026-06-18): **Host capture policy.** Capture mode is bounded by an immutable host `CapturePolicy` seeded into Core state via `session.start`, not chosen freely by the client. Product default DENIES sensitive capture (`rowCapture=full` / `sqlCapture=text`); `v2/diagnostics.setCapture` may only move within the policy and is rejected (`Sts2.InvalidRequest`) when it would exceed it. A development/test composition can permit elevation. `initialize` advertises the allowed maxima. Chosen over leaving capture open so an untrusted or compromised client cannot persist real SQL/row data to the journal in production.

## SPEC-CHANGE entries

- **SPEC-CHANGE-0001** (2026-07-06, STS2-3: `maxCellBytes` honored on the wire; owner-directed, left uncommitted for the mandated human review):
  - **Problem.** The contract advertised `limits.maxCellBytes` but the bound was pinned server-side (the R024 fix hardcoded `Sts2Defaults.MaxCellBytes` at the encode edge); a client could not lower it, no capability advertised honoring it (vscode-mssql `sts2Backend` declares `maxCellBytesHonored: false`), and the as-built truncated wrapper (`{"$t","of","bytes","v"}`) had drifted from the SPEC Â§7.7 shape (`digest` and the `truncatedPrefixBytes` cap were unimplemented).
  - **Options.** (a) Per-page truncated-cell index list on `v2/query.rows` â€” a new envelope field, two sources of truth per page, and nothing for the digest drift. (b) Per-cell `$t:"truncated"` wrapper converged WITH the SPEC (add `digest`, honor the `truncatedPrefixBytes` cap, keep the additive `of`) plus an OPTIONAL `options.maxCellBytes` on `v2/query.execute` and a `maxCellBytesHonored` capability. (c) Silent clamp â€” rejected outright (violates truncation honesty).
  - **Decision: (b).** All wire deltas are additive/optional: `options.maxCellBytes` (absent = prior behavior), `capabilities.maxCellBytesHonored: true`, wrapper fields `of` (already shipped) + `digest` (SPEC-promised). Semantics: absent/0/negative/non-integer means the pinned default (pre-existing behavior; the R024 memory bound is NOT removable by the client); positive values clamp to the pinned default (lower-only). Core normalizes the effective bound into the journaled `driver.queryStart` args so replay stays deterministic; the runner truncates at the encode edge, so every driver (SqlClient, Sqlite, Fake) is covered. Pinned defaults unchanged (`sts2.results.maxCellBytes` 1048576, `sts2.results.truncatedPrefixBytes` 65536 â€” the latter now actually enforced).
  - **Evidence.** Scenario `cell-truncation-max-cell-bytes` activated (200KB string + 100KB blob at a 64-byte bound; wrapper metadata asserted); QueryFlowTests wire tests (bounded pages, digest honesty, exact-bound passthrough, UTF-8 code-point safety incl. astral plane, absent/0 unchanged behavior, clamp + prefix cap); CoreReducerTests option normalization; WireValueEncoderTests digest/boundary/prefix-cap; `initialize-idempotent` pins the capability. Precedent: `database` on `v2/query.complete` (2026-07-05) treated additive optional wire fields as review-safe with SPEC text updated in the same change.
